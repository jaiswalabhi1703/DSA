Do you need plain Spring before Boot?

"Jump to Spring Boot" vs "Without learning Spring"

You can start directly with Spring Boot — Boot hides the XML/config pain that made classic Spring intimidating. But the concepts (IoC, beans, DI, the ApplicationContext) are pure Spring; Boot only automates their wiring. So: jump into Boot to be productive fast, but you must understand the underlying Spring container or auto-configuration feels like magic you can't debug.

WhyBoot is not a new framework — it's Spring with batteries included. Every Boot feature maps to a Spring building block underneath.
PathLearn the container model first (this "Core" tab), then let Boot's auto-config remove the boilerplate. You skip XML, not concepts.

IoC — Inversion of Control

Who creates your objects: you, or the framework?

IoC means you hand object creation and wiring to the framework instead of calling new yourself. The container reads bean definitions, instantiates them, injects dependencies, and manages their lifecycle. DI is the technique that implements IoC.

// WITHOUT IoC — tight coupling, you control creation class OrderService { private PaymentGateway gw = new StripeGateway(); // hard-wired! } // WITH IoC — container injects the dependency @Service class OrderService { private final PaymentGateway gw; OrderService(PaymentGateway gw) { this.gw = gw; } // Spring decides impl }
WhyClasses depend on interfaces, the container picks the implementation → testable, swappable, no hidden new scattered around.
TrapIoC = the principle (don't control your own deps); DI = the implementation (deps are injected). Not synonyms.
💉

Dependency Injection — 3 styles

Constructor, setter, field — and why constructor wins
// 1. CONSTRUCTOR INJECTION ← preferred @Service class OrderService { private final PaymentGateway gw; // final = immutable OrderService(PaymentGateway gw) { this.gw = gw; } // @Autowired optional if single ctor } // 2. SETTER INJECTION — for OPTIONAL deps @Autowired(required = false) public void setCache(Cache cache) { this.cache = cache; } // 3. FIELD INJECTION — concise, but discouraged @Autowired private PaymentGateway gw;
✓ Constructor
  • Fields can be final → immutable
  • Deps guaranteed at construction
  • Fails fast on missing / circular deps
  • Trivial to unit-test (just new + mocks)
✕ Field injection
  • No final — mutable
  • Hides deps; needs reflection to test
  • Lets circular deps sneak in
  • Spring team recommends against it
UseDefault to constructor injection. Setter only for genuinely optional deps. Avoid field injection outside tests.
@Autowired@Qualifier@Primary@Value

The 4 layers: Controller · Service · Repository · DTO

Separation of concerns done the Spring way
// ─── CONTROLLER: HTTP only, no business logic ─── @RestController @RequestMapping("/api/orders") class OrderController { private final OrderService service; OrderController(OrderService s) { this.service = s; } @GetMapping("/{id}") OrderDto get(@PathVariable Long id) { return service.find(id); } } // ─── SERVICE: business logic + transactions ─── @Service class OrderService { private final OrderRepository repo; OrderService(OrderRepository r) { this.repo = r; } @Transactional(readOnly = true) OrderDto find(Long id) { Order o = repo.findById(id).orElseThrow(() -> new OrderNotFound(id)); return OrderDto.from(o); // map entity → DTO } } // ─── REPOSITORY: persistence, generated by Spring Data ─── interface OrderRepository extends JpaRepository<Order, Long> {} // ─── DTO: the wire shape, decoupled from the entity ─── record OrderDto(Long id, String customer) { static OrderDto from(Order o) { return new OrderDto(o.getId(), o.getCustomer()); } }
Why layerEach layer has one job: Controller = HTTP, Service = business rules + tx boundary, Repository = data, DTO = API contract. Swap any layer without touching the others.
DTO trapReturning entities directly → LazyInitializationException (session closed) or serializing the whole object graph. DTOs prevent both and hide sensitive fields.

Exception Handling — @ControllerAdvice

One place for all your error responses
@RestControllerAdvice // global — applies to all controllers class ApiExceptionHandler { @ExceptionHandler(OrderNotFound.class) @ResponseStatus(HttpStatus.NOT_FOUND) ApiError notFound(OrderNotFound ex) { return new ApiError("ORDER_NOT_FOUND", ex.getMessage()); } @ExceptionHandler(MethodArgumentNotValidException.class) @ResponseStatus(HttpStatus.BAD_REQUEST) ApiError invalid(MethodArgumentNotValidException ex) { return new ApiError("VALIDATION", ex.getBindingResult().getFieldError().getDefaultMessage()); } }
WhyKeeps controllers free of try/catch, guarantees a consistent error contract, maps domain exceptions → correct HTTP codes in one file.
ModernSpring 6 / Boot 3 support RFC 7807 ProblemDetail as a standardized error body. Extend ResponseEntityExceptionHandler to override framework errors too.

Filters vs Interceptors

Two hook points — and when to use each
Filter (Servlet)Interceptor (Spring MVC)
LayerServlet container, before DispatcherServletInside Spring MVC
Knows handler?NoYes (which controller)
HooksdoFilterpreHandle/postHandle/afterCompletion
Best forAuth, CORS, compression, logging raw reqAuth checks needing handler, timing, MDC
@Component class RequestLogFilter implements Filter { @Override public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) throws IOException, ServletException { long t = System.currentTimeMillis(); chain.doFilter(req, res); // continue the chain log.info("took {}ms", System.currentTimeMillis() - t); } } class AuthInterceptor implements HandlerInterceptor { @Override public boolean preHandle(HttpServletRequest req, HttpServletResponse res, Object handler) { return isValid(req.getHeader("X-Token")); // false → request stops here } }

Pagination & Sorting

Never return an unbounded list
@GetMapping Page<OrderDto> list(Pageable pageable) { // ?page=0&size=20&sort=amount,desc return repo.findAll(pageable).map(OrderDto::from); } // Page = content + totalElements + totalPages (runs a count query) // Slice = content + hasNext only (no count → cheaper, infinite scroll)
ScaleAlways paginate. Use Slice for infinite scroll (skips the expensive count(*)). For huge offsets prefer keyset/cursor pagination over OFFSET.
🔗

Cascading & Connection Pooling

Propagating operations + reusing DB connections
@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true) private List<OrderItem> items; // save/delete order → cascades to items
CascadeTypeEffect
PERSISTSave parent → save children
MERGEUpdate parent → update children
REMOVEDelete parent → delete children
ALLAll of the above
Cascade trapCascadeType.ALL + orphanRemoval on the wrong relation = accidental mass deletes. Apply cascades only from the true "owning" parent.
PoolingBoot ships HikariCP (fastest pool) by default. A pool reuses a fixed set of DB connections instead of opening one per request — tune spring.datasource.hikari.maximum-pool-size to match DB limits. Opening connections per-request kills throughput.
💾

Database Support & ACID

H2 for dev, MySQL/Postgres for prod
# H2 (in-memory) — zero config, great for tests/dev spring.datasource.url=jdbc:h2:mem:testdb spring.jpa.hibernate.ddl-auto=create-drop # MySQL / PostgreSQL — production spring.datasource.url=jdbc:postgresql://db:5432/shop spring.datasource.username=app spring.datasource.password=${DB_PASSWORD} # from env, not committed spring.jpa.hibernate.ddl-auto=validate # never 'update' in prod

ACID — the guarantees a transaction gives:

  • Atomicity — all steps commit or none do (rollback on failure)
  • Consistency — constraints/invariants hold before & after
  • Isolation — concurrent txns don't corrupt each other
  • Durability — committed data survives a crash
ddl-autocreate-drop for tests, validate or none for prod. update in prod silently mutates schema and can corrupt data — use Flyway/Liquibase migrations instead.

Transaction Management & Isolation

@Transactional + the proxy gotcha + isolation levels
@Service class TransferService { @Transactional // commit on success, rollback on RuntimeException void transfer(Long from, Long to, BigDecimal amt) { accounts.debit(from, amt); accounts.credit(to, amt); // if this throws, debit rolls back too } @Transactional(isolation = Isolation.REPEATABLE_READ, timeout = 5) void criticalUpdate() { ... } }
IsolationPreventsCost
READ_UNCOMMITTEDnothing (dirty reads OK)fastest
READ_COMMITTED (default)dirty readslow
REPEATABLE_READ+ non-repeatable readsmedium
SERIALIZABLE+ phantom readshighest
#1 gotchaIt's an AOP proxy: a method calling this.otherTx() in the same class bypasses the proxy → no transaction. Only public methods are advised, and by default only RuntimeException/Error roll back (use rollbackFor for checked).
PropagationREQUIRED (default), REQUIRES_NEW (independent tx — audit logs that persist even if outer rolls back), NESTED (savepoints).
High contentionPrefer optimistic locking (@Version) over heavy isolation — fail-and-retry beats locking rows under load.

JDBC — JdbcTemplate & RowMapper

"Not needed" with JPA, but know it exists

With Spring Data JPA you rarely touch raw JDBC. But JdbcTemplate is the escape hatch for complex/native SQL, bulk ops, or when an ORM is overkill.

@Repository class OrderJdbcDao { private final JdbcTemplate jdbc; OrderJdbcDao(JdbcTemplate jdbc) { this.jdbc = jdbc; } Order find(Long id) { return jdbc.queryForObject( "select id, customer from orders where id = ?", (rs, rowNum) -> new Order(rs.getLong("id"), rs.getString("customer")), // RowMapper id); } }
WhenJPA for object-graph CRUD (95%); JdbcTemplate for hand-tuned SQL, reporting queries, or batch inserts where Hibernate adds overhead.

ORM Fundamentals & SessionFactory

Objects ↔ rows, without writing SQL

ORM (Object-Relational Mapping) maps Java objects to DB rows so you work with entities, not SQL. Hibernate is the JPA implementation Boot uses by default. Key objects:

  • SessionFactory — heavyweight, thread-safe, one per app. Built once at startup, holds config + connection pool. (JPA equivalent: EntityManagerFactory.)
  • Session — lightweight, not thread-safe, one per unit of work/request. The first-level cache lives here. (JPA: EntityManager.)
  • Transaction — wraps a unit of work.
TrapSessionFactory = expensive, create once. Session = cheap, create per request. Never share a Session across threads.
🌀

Entity Lifecycle States

Transient → Persistent → Detached → Removed
StateMeaningIn session? In DB?
TransientJust new-ed, never savedNo / No
PersistentManaged — changes auto-flush (dirty checking)Yes / Yes
DetachedWas persistent, session closedNo / Yes
RemovedMarked for deleteYes / pending delete
Dirty checkingOn a persistent entity, just changing a field auto-generates an UPDATE at flush — no explicit save() needed inside a transaction.
LazyInitTouching a lazy field on a detached entity (session closed) throws LazyInitializationException — the #1 reason to map to DTOs inside the tx.

Lazy vs Eager + the N+1 Problem

The single most-asked Hibernate performance question
MappingDefault fetch
@ManyToOne / @OneToOneEAGER
@OneToMany / @ManyToManyLAZY
// N+1: loop 100 orders, each lazily touches getUser() List<Order> orders = repo.findAll(); // 1 query for (Order o : orders) sysout(o.getUser().getName()); // + 100 queries 😱 // FIX 1 — fetch join (JPQL) @Query("select o from Order o join fetch o.user") List<Order> findAllWithUser(); // FIX 2 — entity graph @EntityGraph(attributePaths = "user") List<Order> findAll(); // FIX 3 — batch fetching @BatchSize(size = 50) // loads users in chunks of 50, not 1-by-1
Best practiceMake everything LAZY (even @ManyToOne(fetch = LAZY)) and fetch explicitly per use case. EAGER everywhere loads half the DB on every query.

HQL vs Criteria API

Two ways to query beyond derived methods
// HQL / JPQL — queries ENTITIES, not tables (string-based) List<Order> orders = em.createQuery( "select o from Order o where o.amount > :min order by o.amount desc", Order.class) .setParameter("min", new BigDecimal("100")).getResultList(); // Criteria API — type-safe, programmatic, good for DYNAMIC queries CriteriaBuilder cb = em.getCriteriaBuilder(); CriteriaQuery<Order> q = cb.createQuery(Order.class); Root<Order> root = q.from(Order.class); q.select(root).where(cb.gt(root.get("amount"), 100));
WhenHQL for fixed, readable queries. Criteria for queries built dynamically from optional filters (search forms) where string concatenation would be ugly and unsafe.

Basics: AuthN vs AuthZ & the Filter Chain

Who are you, vs what may you do

Authentication (AuthN) = verify identity (login, JWT, OAuth). Authorization (AuthZ) = verify permissions (roles, scopes). Spring Security is a chain of servlet filters in front of your app; each filter handles one concern (CORS, CSRF, auth, etc.).

@Configuration @EnableWebSecurity class SecurityConfig { @Bean SecurityFilterChain chain(HttpSecurity http) throws Exception { return http .csrf(c -> c.disable()) // stateless API .authorizeHttpRequests(a -> a .requestMatchers("/api/public/**").permitAll() .requestMatchers("/api/admin/**").hasRole("ADMIN") .anyRequest().authenticated()) .sessionManagement(s -> s.sessionCreationPolicy(STATELESS)) .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class) .build(); } @Bean PasswordEncoder encoder() { return new BCryptPasswordEncoder(); } }
Method security@EnableMethodSecurity then @PreAuthorize("hasRole('ADMIN')") or @PreAuthorize("#userId == authentication.principal.id").
PasswordsBCrypt/Argon2 are adaptive + salted — slow by design. Never MD5/SHA-256, never plaintext.

JWT & Stateless Authentication

Why microservices love tokens

A JWT = header.payload.signature, base64url-encoded and signed (HMAC or RSA). The server validates the signature instead of a DB session lookup → stateless & horizontally scalable.

1Login → verify creds→ 2Server signs JWT→ 3Client stores & sends in Authorization header→ 4Filter validates signature + expiry→ 5Set SecurityContext
✓ JWT pros
  • Stateless — no session store
  • Scales horizontally trivially
  • Carries claims (roles) inline
  • Cross-service / SSO friendly
✕ JWT cons
  • Hard to revoke before expiry
  • Bigger than a session id
  • Payload readable (no secrets!)
  • Needs refresh-token strategy
Session vs JWTServer session for a monolith needing easy revocation. JWT for stateless REST/microservices/mobile. Mitigate revocation with short expiry + refresh tokens + a denylist.
🔑

OAuth2 — login, social, resource server

Delegated auth without handling passwords

OAuth2 lets users log in via a third party (Google, GitHub) without your app ever seeing their password. Spring Security has first-class support:

  • OAuth2 Login / Social login — spring-boot-starter-oauth2-client; configure provider client-id/secret, Spring handles the redirect dance.
  • Resource Server — spring-boot-starter-oauth2-resource-server; your API validates incoming JWT/opaque tokens issued by an auth server (Keycloak, Auth0, Okta).
# Resource server — just point at the issuer, Spring validates tokens spring.security.oauth2.resourceserver.jwt.issuer-uri=https://auth.example.com
WhyOffloads identity to a dedicated provider — better security, SSO across apps, no password storage. The four roles: Resource Owner (user), Client (your app), Authorization Server, Resource Server (your API).

API Design Principles

What separates a clean API from a messy one
  • Nouns, not verbs — GET /orders/42, not /getOrder?id=42.
  • Plural collections — /orders, /orders/42/items (nesting for relationships).
  • Versioning — /api/v1/orders so you can evolve without breaking clients.
  • Filtering/paging via query params — ?status=PAID&page=0&size=20.
  • Consistent errors — same JSON error shape everywhere (via @ControllerAdvice).
  • HATEOAS (optional) — responses include links to related actions.
WhyPredictable URLs + correct verbs + stable contracts mean clients can be written once and not rewritten every release.

JSON/XML & Swagger / OpenAPI

Content negotiation + self-documenting APIs

The questions a Spring/Boot round actually opens with. Lead with the one-liner, give an example, anticipate the follow-up.
01Difference between Spring and Spring Boot?Must know
Spring is the core IoC/DI framework (+ MVC, Data, AOP) needing heavy manual config. Spring Boot sits on top adding auto-configuration, starter dependencies, an embedded server, and Actuator. Boot isn't a replacement — it's an opinionated layer that configures Spring for you so you ship faster.
02IoC vs Dependency Injection?Must know
IoC is the principle: hand object creation/wiring to the framework. DI is the pattern that achieves it — dependencies are injected (constructor/setter/field) rather than new-ed. IoC = goal, DI = mechanism.
03Constructor vs field injection — which and why?Must know
Constructor injection — allows final immutable fields, guarantees deps at construction, fails fast on missing/circular deps, and is trivially testable without Spring. Field injection hides deps and needs reflection to test. Setter is fine for optional deps.
04How does auto-configuration work?Must know
@EnableAutoConfiguration loads candidate config classes guarded by @Conditional annotations (@ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty). Boot inspects the classpath and your beans, applying defaults only when conditions match and you haven't defined that bean. Override by declaring your own bean; debug with --debug.
05Default bean scope? Are singletons thread-safe?Must know
Default is singleton — one instance per container, shared across all threads. So singletons must be stateless; never store per-request mutable state in fields. Spring singleton ≠ GoF singleton — it's per-container, not per-JVM.
06Why might @Transactional silently not work?Must know
It's proxy-based, so: (1) self-invocation via this. bypasses the proxy; (2) method isn't public; (3) a checked exception was thrown (default rollback is only on RuntimeException/Error — use rollbackFor); (4) the call originates outside a Spring bean.
07What is the N+1 problem and how do you fix it?Must know
Fetching a parent list then lazily loading each child = 1 + N queries. Fix with a fetch join (JOIN FETCH), an @EntityGraph, or batch fetching (@BatchSize). Root cause: make associations LAZY and fetch explicitly per use case.
08@Component vs @Bean?Good to know
@Component is class-level, auto-detected by scanning — for your own classes. @Bean is a method in a @Configuration class you write to build/configure an object — for third-party types you can't annotate or when construction needs logic.
09How does Spring AOP work under the hood?Good to know
Via runtime proxies: JDK dynamic proxies if the bean implements an interface, else CGLIB subclassing. A BeanPostProcessor wraps the target; callers get the proxy which runs advice around the real method. That's why @Transactional/@Cacheable/@Async share the self-invocation limitation.
10Singleton vs prototype — and the injection trap?Good to know
Singleton = one per container; prototype = new each request. Injecting a prototype into a singleton resolves it once, so you keep the same instance. Fix with @Lookup, ObjectProvider<T>, or a scoped proxy.
11Lazy vs Eager loading — defaults?Good to know
@ManyToOne/@OneToOne default EAGER; @OneToMany/@ManyToMany default LAZY. Best practice: make everything LAZY and fetch-join where needed. Touching a lazy field after the session closes throws LazyInitializationException.
12PUT vs PATCH vs POST?Good to know
POST creates (non-idempotent). PUT replaces the whole resource (idempotent — repeat = same state). PATCH partially updates only the fields sent. Idempotency matters for safe retries.
13Session vs JWT for auth?Good to know
Sessions store state server-side (easy revoke, needs sticky/shared store to scale). JWT is stateless — verify a signature, no lookup — scales horizontally and suits microservices, but is hard to revoke before expiry. Mitigate with short access tokens + refresh tokens + denylist.
14L1 vs L2 cache in Hibernate?Good to know
L1 is per-Session, always on — same entity loaded twice in one session hits cache. L2 is per-SessionFactory (app-wide), opt-in, backed by EhCache/Hazelcast — shared across sessions for read-mostly data, at the cost of invalidation complexity.
15@SpringBootTest vs slice tests?Good to know
@SpringBootTest boots the full context — slow, for integration. Slice tests (@WebMvcTest, @DataJpaTest) load one layer for fast focused tests, mocking the rest with @MockBean. Default to slices; reserve full context for end-to-end paths.
16How does Spring resolve two beans of the same type?Good to know
By default it throws NoUniqueBeanDefinitionException. Resolve with @Primary (default winner) or @Qualifier("name") at the injection point. Zero candidates instead throws NoSuchBeanDefinitionException — usually a missing stereotype or component scan.

Core / IoC

  • IoC = principle, DI = mechanism
  • Constructor injection + final
  • ApplicationContext over BeanFactory
  • Singletons must be stateless
  • @PostConstruct / @PreDestroy hooks

Web (MVC/REST)

  • Controller → Service → Repo → DTO
  • @RestController = @Controller + @ResponseBody
  • @ControllerAdvice for global errors
  • Filter (container) vs Interceptor (MVC)
  • Return DTOs, never entities

Data / JPA

  • JpaRepository = derived queries free
  • Make associations LAZY, fetch-join
  • N+1 → JOIN FETCH / @EntityGraph
  • @Transactional = proxy (self-call trap)
  • Always paginate lists

Hibernate

  • SessionFactory once, Session per request
  • Lifecycle: transient→persistent→detached
  • L1 per-session, L2 app-wide opt-in
  • Dirty checking auto-updates
  • HQL fixed, Criteria dynamic

Security

  • AuthN (who) vs AuthZ (what)
  • SecurityFilterChain bean (Boot 3)
  • BCrypt for passwords, never plaintext
  • JWT = stateless, hard to revoke
  • OAuth2 resource server = validate tokens

REST / Build / Advanced

  • GET safe, PUT idempotent, POST not
  • Fat JAR + embedded Tomcat
  • Config from env, not the build
  • Actuator: health/info/metrics (secure it)
  • WebClient over RestTemplate
One-line close: Spring gives you an IoC container; Boot makes it production-ready out of the box. MVC, Data, Hibernate, Security, REST and the advanced features are all modules that plug into the same bean container and are wired the same way. Master the container and the rest is vocabulary.
